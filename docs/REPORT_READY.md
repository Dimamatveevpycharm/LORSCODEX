## Задание

В рамках курсовой работы необходимо реализовать учебный мини-компилятор языка программирования с последующим выпуском объектного файла под целевую архитектуру и демонстрацией его выполнения через внешний `main.c`.

Для данного проекта реализован вариант со следующими параметрами:

- Минимальный набор конструкций: арифметические и логические операции, `if-else`, `for`, типы `int`, `float`, функции.
- Стиль синтаксиса: **Go-style** (блоки `{ ... }`, ключевое слово `func`, `var`).
- Дополнительная конструкция: **тернарный оператор `?:`**.
- Целевая архитектура: **RISC-V** (`riscv64-unknown-linux-gnu`).

В проекте выполнены все обязательные этапы:

1. Лексический анализ на Flex.
2. Синтаксический анализ на GNU Bison.
3. Отдельный семантический анализ.
4. Перевод AST в LLVM IR.
5. Выпуск объектного файла.
6. Линковка с `runtime/main.c`.
7. Запуск результата в эмуляторе `qemu-riscv64`.
8. Создание тестового набора и проверка корректности.

## 1. Лексическая спецификация

Лексический анализатор реализован в файле `src/lexer.l`.

Распознаются:

- Ключевые слова: `func`, `var`, `if`, `else`, `for`, `return`, `int`, `float`, `bool`, `void`, `true`, `false`.
- Идентификаторы: шаблон `[A-Za-z_][A-Za-z0-9_]*`.
- Литералы:
  - целые (`INT_LIT`);
  - вещественные (`FLOAT_LIT`, включая экспоненциальную форму).
- Операторы:
  - арифметические: `+ - * / %`;
  - сравнения: `== != < <= > >=`;
  - логические: `&& || !`;
  - тернарные: `? :`;
  - присваивание: `=`.
- Разделители: `, ; ( ) { }`.
- Комментарии:
  - однострочные `// ...`;
  - многострочные `/* ... */`.

Особенности реализации:

- Ведётся позиционная информация (`line`, `column`) для точной диагностики ошибок.
- Для каждого токена перед возвратом выставляются координаты через `%locations` и `yylloc`.

## 2. Диаграмма разработанных правил грамматики (в текстовом виде)

Ниже приведена текстовая структурная диаграмма грамматики (вместо графической схемы):

1. `program`
   - состоит из `function_list`.
2. `function_list`
   - одна или более `function_decl`.
3. `function_decl`
   - `func IDENT ( param_list_opt ) type_spec block`.
4. `block`
   - `{ stmt_list }`.
5. `stmt_list`
   - ноль или более `statement`.
6. `statement`
   - `var_decl ;`
   - `assign_stmt ;`
   - `expression ;`
   - `return_stmt ;`
   - `block`
   - `if expression block`
   - `if expression block else block`
   - `for expression block`
   - `for for_init ; expression ; for_post block`
7. `expression`
   - литералы, идентификаторы, вызовы функций;
   - унарные операции (`-`, `!`);
   - бинарные операции с приоритетами;
   - тернарный оператор `condition ? expr1 : expr2`.

Текстовая схема конвейера синтаксического уровня:

`source` -> `tokens (Flex)` -> `Bison grammar` -> `AST nodes` -> `Program`

## 4. Описание семантических правил языка

Семантический анализ реализован отдельно в `src/Sema.cpp`.

### 4.1. Основные правила

1. Имена:
   - запрет повторного объявления переменной в одном scope;
   - разрешён доступ к переменным внешнего scope;
   - ошибка при использовании необъявленного имени.
2. Типы:
   - поддерживаются `int`, `float`, `bool`, `void`;
   - единственное неявное преобразование: `int -> float`;
   - запрещено неявное `float -> int`.
3. Выражения:
   - арифметика только для числовых типов;
   - `%` только для `int`;
   - логические операторы только для `bool`;
   - сравнения возвращают `bool`.
4. Управляющие конструкции:
   - условия `if` и `for` должны иметь тип `bool`.
5. Функции:
   - проверка сигнатур;
   - проверка количества и типов аргументов;
   - проверка корректности `return` по типу функции.
6. Тернарный оператор:
   - условие должно быть `bool`;
   - ветви должны быть совместимы по типу.

### 4.2. Структуры данных

- Глобальная таблица функций.
- Стек областей видимости для локальных переменных.

## 6. Описание процесса перевода AST в IR и из IR в obj

Генерация кода реализована в `src/Codegen.cpp`.

### 6.1. AST -> LLVM IR

1. Создаётся `LLVMContext`, `Module`, `IRBuilder`.
2. Для каждой функции:
   - объявляется сигнатура;
   - создаётся `entry`-блок;
   - параметры раскладываются в `alloca`;
   - генерируются инструкции для всех statement/expr.
3. Генерация конструкций:
   - `if/else` -> набор `BasicBlock` с переходами;
   - `for` -> `cond/body/post/exit` блоки;
   - `&&`, `||` -> short-circuit через ветвления и `PHI`;
   - `?:` -> ветви + `merge` + `PHI`.
4. Выполняется `verifyFunction`.
5. После генерации всего модуля выполняется `verifyModule`.

### 6.2. LLVM IR -> объектный файл

1. Инициализируются target-компоненты LLVM.
2. Выбирается target по triple (`riscv64-unknown-linux-gnu`).
3. Создаётся `TargetMachine`.
4. Через `addPassesToEmitFile` выпускается `.o`.

В проекте подтверждена генерация RISC-V объекта:

`file build/demo_entry.o` -> `ELF 64-bit ... UCB RISC-V ...`

## 9. Описание процесса использования компилятора

### 9.1. Сборка компилятора

```bash
cmake -S . -B build
cmake --build build -j
```

### 9.2. Генерация IR и объектного файла

```bash
./build/lorsc tests/integration/demo_entry.lors -o build/demo_entry.o --emit-llvm build/demo_entry.ll
```

### 9.3. Проверка target triple в IR

```bash
grep -n "target triple" build/demo_entry.ll
```

Ожидается: `riscv64-unknown-linux-gnu`.

### 9.4. Линковка с `runtime/main.c`

В текущей WSL-среде рабочая команда:

```bash
clang --target=riscv64-unknown-linux-gnu -fuse-ld=lld -static runtime/main.c build/demo_entry.o -o build/demo_riscv
```

Комментарий по среде:

- Вариант линковки без `-fuse-ld=lld` в этой среде может падать с ошибкой типа `unrecognised emulation mode: elf64lriscv`.
- Статическая линковка (`-static`) устраняет проблему отсутствующего RISC-V dynamic loader при запуске в QEMU.

### 9.5. Запуск результата

```bash
qemu-riscv64 build/demo_riscv
```

Фактический вывод:

```text
entry() = 8
```
