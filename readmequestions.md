# Ответы на вопросы по проекту LORSC

Ниже для каждого вопроса даны два варианта ответа:

1. На уровне ВУЗа, но максимально понятно.
2. Более упрощённо для быстрого понимания.

## 1. Как в коде реализованы правила грамматики? Что делает код с `$$` (`semantic actions`)?

### Вариант 1. На уровне ВУЗа

Правила грамматики реализованы в файле `src/parser.y` с помощью `Bison`. Каждое правило описывает, из каких токенов и нетерминалов может состоять конструкция языка. Например, правило

```bison
expression
  : expression PLUS expression {
      $$ = new lorsc::BinaryExpr(MAKE_LOC(@2), lorsc::BinaryOpKind::Add, $1, $3);
    }
```

означает, что выражение может быть суммой двух подвыражений.

Код в фигурных скобках после правила называется `semantic action`, то есть семантическое действие. Оно выполняется в момент, когда парсер успешно распознал это правило. В нашем проекте это действие обычно создаёт узел абстрактного синтаксического дерева (AST, abstract syntax tree).

Обозначения такие:

- `$1`, `$2`, `$3` — значения компонентов правила справа.
- `$$` — значение, которое будет присвоено всему правилу слева.

Например, в правиле для сложения:

- `$1` — левое выражение,
- `$2` — сам оператор `+`,
- `$3` — правое выражение,
- `$$` — новый AST-узел `BinaryExpr`, представляющий всё выражение целиком.

То есть `$$` не "что-то магическое", а результат разбора текущей грамматической конструкции. В проекте через `$$` строятся `Program`, `FunctionDecl`, `BlockStmt`, `IfStmt`, `ForStmt`, `BinaryExpr`, `UnaryExpr` и другие узлы дерева.

### Вариант 2. Упрощённо

Грамматика описана в `src/parser.y` как набор правил "из чего состоит язык". Когда `Bison` узнаёт какую-то конструкцию, он выполняет код после правила.

`$$` — это "итог" текущего правила.

`$1`, `$2`, `$3` — это части, из которых это правило собралось.

Например, для `a + b`:

- `$1` — `a`,
- `$3` — `b`,
- `$$` — готовый узел "сложение `a` и `b`".

То есть через `$$` парсер постепенно собирает дерево программы.

## 2. Как и где происходит генерация кода под целевую платформу?

### Вариант 1. На уровне ВУЗа

Генерация кода происходит в файле `src/Codegen.cpp`, в классе `CodeGenerator`. Этот модуль получает уже готовое AST после синтаксического и семантического анализа и преобразует его в `LLVM IR` (Low Level Virtual Machine Intermediate Representation, промежуточное представление LLVM).

Основной вход в этап генерации — метод `generate(Program& program)`. Сначала он объявляет функции через `declareFunctions`, затем определяет их тела через `defineFunction`.

Дальше для каждой конструкции используются отдельные методы:

- `codegenExpr` — генерация выражений,
- `codegenIf` — генерация ветвления,
- `codegenFor` — генерация цикла,
- `codegenReturn` — генерация возврата,
- `codegenVarDecl` и `codegenAssign` — работа с переменными.

Сначала строится `LLVM IR`, а уже потом этот IR переводится в код конкретной архитектуры. Привязка к платформе задаётся через `target triple`, который по умолчанию равен `riscv64-unknown-linux-gnu`.

Выбор целевой платформы и создание backend-настроек происходят в методе `createTargetMachine()`. Там вызываются:

- `llvm::TargetRegistry::lookupTarget(...)`,
- `target->createTargetMachine(...)`.

После этого:

- `emitIR(...)` сохраняет текстовый `.ll`,
- `emitAssembly(...)` генерирует `.s`,
- `emitObject(...)` генерирует объектный файл `.o`.

Именно поэтому можно сказать, что проект не генерирует RISC-V инструкции вручную, а сначала строит `LLVM IR`, а затем использует backend LLVM для получения кода под нужную платформу.

### Вариант 2. Упрощённо

Генерация кода находится в `src/Codegen.cpp`.

Схема такая:

1. Парсер строит дерево программы.
2. Семантика проверяет, что программа корректна.
3. `CodeGenerator` превращает дерево в `LLVM IR`.
4. LLVM уже сам переводит это в код под `RISC-V`.

То есть наш код в основном строит промежуточное представление, а низкоуровневый код под целевую архитектуру делает backend LLVM.

## 3. Объясните роль `flex`/`re2c` и `bison` в проекте.

### Вариант 1. На уровне ВУЗа

В этом проекте реально используется `Flex`, а не `re2c`.

`Flex` применяется в файле `src/lexer.l` и отвечает за лексический анализ. Он читает поток символов и превращает его в поток токенов: ключевые слова, идентификаторы, литералы, операторы, скобки и так далее. Например, именно `Flex` различает `func`, `if`, `return`, `123`, `3.14`, `+`, `==`.

`Bison` используется в файле `src/parser.y` и отвечает за синтаксический анализ. Он получает токены от лексера и проверяет, соответствует ли их последовательность грамматике языка. Если соответствует, `Bison` выполняет semantic actions и строит AST.

Иначе говоря:

- `Flex` отвечает на вопрос "какие здесь слова и символы языка?",
- `Bison` отвечает на вопрос "как из этих токенов складываются корректные конструкции языка?".

`re2c` — это альтернативный генератор лексеров, но в данном проекте он не используется.

### Вариант 2. Упрощённо

`Flex` режет текст программы на маленькие части: слова, числа, знаки операций.

`Bison` из этих частей собирает уже полноценные конструкции языка: функции, выражения, `if`, `for`, `return`.

`re2c` здесь не используется, он просто возможная альтернатива `Flex`.

## 4. Объясните, зачем нужен файл с кодом абстрактного синтаксического дерева.

### Вариант 1. На уровне ВУЗа

Абстрактное синтаксическое дерево описано в `include/lorsc/AST.h`. Оно нужно как внутренняя структура данных, в которой программа хранится уже после синтаксического разбора, но ещё до генерации машинного кода.

Парсеру неудобно сразу генерировать код, потому что сначала нужно понять структуру программы. Поэтому он строит AST: отдельные классы для литералов, переменных, бинарных операций, вызовов функций, блоков, циклов, условий, возврата и всей программы целиком.

AST в проекте решает несколько задач:

- хранит структуру программы в удобном виде;
- отделяет синтаксический анализ от семантического;
- позволяет семантическому анализу проходить по дереву и вычислять типы;
- позволяет этапу codegen независимо проходить по тем же узлам и строить `LLVM IR`.

То есть AST — это центральное представление программы, через которое связаны parser, sema и codegen.

### Вариант 2. Упрощённо

AST — это дерево программы.

Оно нужно, чтобы после разбора у нас была не просто строка текста, а удобная структура:

- вот функция,
- вот её тело,
- вот цикл,
- вот выражение,
- вот операция сложения.

Потом это дерево используют и проверка семантики, и генерация кода. Без него пришлось бы много раз заново разбирать текст.

## 5. Как реализован приоритет арифметических операций?

### Вариант 1. На уровне ВУЗа

Приоритет операций задан в `src/parser.y` через декларации приоритетов `Bison`:

```bison
%right QUESTION COLON
%left OR
%left AND
%left EQ NEQ
%left LT LTE GT GTE
%left PLUS MINUS
%left STAR SLASH PERCENT
%right NOT UMINUS
```

Чем ниже объявление, тем выше приоритет. Поэтому:

- `*`, `/`, `%` имеют приоритет выше, чем `+` и `-`;
- сравнения идут ниже арифметики;
- логические операции идут ещё ниже;
- унарный минус и `!` имеют высокий приоритет;
- тернарный оператор `?:` вынесен отдельно.

Кроме того, для унарного минуса используется запись:

```bison
| MINUS expression %prec UMINUS
```

Это нужно, чтобы `-x` не путался с бинарным вычитанием `a - b`, и чтобы для него действовал отдельный, более высокий приоритет.

Итог: приоритет арифметических операций задаётся не вручную в кодогенерации, а на этапе грамматики, ещё во время синтаксического анализа.

### Вариант 2. Упрощённо

Приоритеты заданы прямо в `parser.y`.

Из-за этого парсер заранее знает, что:

- сначала надо считать `*` и `/`,
- потом `+` и `-`,
- потом сравнения,
- потом логические операции.

Поэтому выражение `2 + 3 * 4` разбирается как `2 + (3 * 4)`, а не `(2 + 3) * 4`.

## 6. Что поддерживает семантический анализ в вашем проекте?

### Вариант 1. На уровне ВУЗа

Семантический анализ реализован в `src/Sema.cpp`, в классе `SemanticAnalyzer`. Он работает уже после построения AST и проверяет не форму программы, а её смысловую корректность.

В проекте он поддерживает:

- регистрацию функций и проверку повторных объявлений функций;
- проверку параметров функций, в том числе запрет `void` как типа параметра;
- области видимости переменных через стек `scope`;
- проверку повторного объявления переменной в одной области видимости;
- проверку использования необъявленных переменных;
- проверку присваиваний;
- проверку типов и вычисление `inferredType` для выражений;
- проверку вызовов функций: существование функции, число аргументов, типы аргументов;
- проверку `if` и `for`: условие должно иметь тип `bool`;
- проверку `return`: соответствие типу функции;
- проверку тернарного оператора `?:`;
- проверку того, что невозвращающая `void` функция и невозвращающая не-`void` функция ведут себя правильно;
- проверку всех путей возврата для не-`void` функций.

Также в проекте задано правило неявного преобразования типов: разрешено только `int -> float`.

### Вариант 2. Упрощённо

Семантический анализ проверяет, что программа не просто "похожа по форме" на правильную, а реально логически корректна.

Он умеет проверять:

- объявлена ли переменная;
- не объявили ли одну и ту же переменную дважды;
- правильно ли вызывается функция;
- совпадают ли типы в присваивании и `return`;
- что в `if` и `for` стоит `bool`;
- что у выражений корректные типы;
- что функция, которая должна что-то вернуть, действительно возвращает значение.

## 7. Как обрабатываются ошибки синтаксиса?

### Вариант 1. На уровне ВУЗа

Ошибки синтаксиса обрабатываются в парсере `Bison`, в файле `src/parser.y`. Для этого включён режим:

```bison
%define parse.error verbose
%locations
```

Это означает:

- парсер формирует более подробные сообщения об ошибках;
- для токенов и конструкций сохраняются координаты в файле.

Функция

```cpp
int yyerror(const char* msg)
```

вызывается `Bison`, когда встречается синтаксическая ошибка. Внутри проекта она передаёт сообщение в `DiagnosticsEngine` через

```cpp
reportParseError(...)
```

Лексер `src/lexer.l` параллельно поддерживает строки и столбцы через `yylloc`, `yylineno`, `yycolumn`, чтобы ошибка была привязана к месту в исходном файле.

Итоговый механизм такой:

1. `Flex` отдаёт токены парсеру.
2. Если последовательность токенов не подходит ни под одно правило грамматики, `Bison` вызывает `yyerror`.
3. Ошибка записывается в систему диагностик с координатами.
4. Пользователь получает понятное сообщение о месте и типе синтаксической ошибки.

### Вариант 2. Упрощённо

Если токены не складываются в правильную конструкцию языка, `Bison` вызывает `yyerror`.

Дальше ошибка передаётся в систему диагностик, где печатается:

- что произошло,
- в каком файле,
- в какой строке и колонке.

То есть проект не просто "падает", а старается выдать понятную синтаксическую ошибку.

## 8. В чем смысл `LLVM IR` и общие принципы его построения?

### Вариант 1. На уровне ВУЗа

`LLVM IR` — это промежуточное представление программы между исходным языком и машинным кодом. В проекте оно нужно как универсальный внутренний формат, на основе которого LLVM уже может сгенерировать код под конкретную архитектуру, например `RISC-V`.

Смысл `LLVM IR` в том, что:

- frontend проекта не обязан вручную генерировать машинные инструкции;
- можно строить одну общую внутреннюю форму программы;
- затем использовать готовую инфраструктуру LLVM для оптимизации, проверки и генерации кода под разные платформы.

В нашем проекте построение IR идёт по AST:

- целая программа становится `Module`;
- каждая функция становится `llvm::Function`;
- блоки управления становятся `BasicBlock`;
- локальные переменные размещаются через `alloca`;
- чтение и запись выполняются через `load` и `store`;
- арифметика строится через `CreateAdd`, `CreateSub`, `CreateMul`, `CreateSDiv`, `CreateFAdd` и другие методы `IRBuilder`;
- ветвления строятся через `CreateCondBr` и `CreateBr`;
- результаты из разных веток объединяются через `PHI`, например для `&&`, `||` и `?:`.

После построения IR проект вызывает `verifyFunction` и `verifyModule`, чтобы убедиться, что представление корректно. Затем через `TargetMachine` из этого IR можно получить:

- текстовый IR (`.ll`),
- ассемблер (`.s`),
- объектный файл (`.o`).

### Вариант 2. Упрощённо

`LLVM IR` — это "промежуточный язык", в который наш компилятор сначала переводит программу.

Зачем это нужно:

- проще не генерировать машинный код вручную;
- LLVM уже умеет хорошо переводить такой IR в код под нужную архитектуру;
- IR удобно проверять и отлаживать.

В проекте сначала строится IR для функций, переменных, операций, условий и циклов, а потом LLVM делает из этого `asm` и объектный файл.

Если совсем коротко: `LLVM IR` — это мост между нашим языком и реальным кодом под процессор.
